## 行動の絶対ルール（自爆防止）

- 【重要】OpenCode（あなた自身）は Node.js 上で動作している。そのため、プロセスを停止するコマンド（`killall node`, `Stop-Process -Name "node"` など）を実行すると自分自身も強制終了してしまう。
- Node.js プロセスを停止する必要がある場合は、決して一括キルを使わず、ポート番号やPIDを指定して個別に停止すること。
- 【重要】**破壊的な削除コマンド（`rm -rf`, `Remove-Item -Recurse -Force`, `rd /s /q` など）の実行は厳禁。**
  - プロジェクトルート（`/home/otr/Repos/x-search-mcp`）全体や絶対パスのルート（`/`）を指定した削除は絶対に実行してはいけない。
  - ファイル削除は、自分が作成した一時ファイルのみ、1つずつ個別に行う。大規模な「整理」や「一括クリーンアップ」は決して行わず、ユーザーに確認すること。

## 秘密情報の保護（Zero-Trust Secret Protection）

**`.env` ファイルを直接編集・新規作成してはいけない。**
- シークレット（APIキー、トークン、パスワードなど）をコードや `.env` に直接書き込んでハードコードすることは絶対に禁止。
- 環境変数の検証や設定変更が必要な場合は、取得ロジック（`process.env` や `os.getenv`）のみを記述し、実際の値の設定はユーザーに手動で依頼すること。
- AIがReadツールのブロックリストを bash 経由（`cat`, `grep` 等）でバイパスして .env を読む行為も禁止。ツールレベルの制限に依存せず、そもそもアクセスしようとしないこと。

## Julesとの連携および役割分担方針

- 【重要】実作業（新規機能の実装、コーディング、大規模リファクタリング、テストの自動生成、Linter/型エラーの一括修正など）は、原則として無料で自律稼働する外部AIエージェント「Jules」にすべて依頼（丸投げ）する。
- 自身（OpenCode）は、APIコスト削減のため、以下の「司令塔（ディレクター）および検証・ペアプロ相手」としての付加価値の高い役割に集中する：
  - ユーザーとの作戦会議、機能の要件定義、アーキテクチャの設計
  - Julesが1回で完璧に実装できるようにするための、詳細で具体的な「チェック指示書（Issue）」の作成と `jules` ラベル付与
  - Julesが作成したPRのコードレビュー、およびローカル環境での入念な動作確認・テストの実行
  - Julesでは解決できない超難解な不具合のデバッグや、ローカル環境に密着した高度なペアプロ作業

### 【必須チェック】Julesへの依頼フロー
- **コード変更タスクでは、作業開始前に必ず「このタスクはJulesに依頼すべきか？」を自問する**
- ファイル編集・テスト追加・リファクタリング・バグ修正は**原則Julesに依頼**
- 自身が直接実装していいのは：設定ファイルの軽微な修正、1行コメント追加、緊急時のホットフィックスのみ

## プロセスの起動・管理

### 環境に応じた永続化戦略

- **Windows (PowerShell 5.1):** `Start-Process`, `Start-Job`, `Start-Process -NoNewWindow` の直接起動はすべて失敗する（Node.js非対話型セッションの制約）。回避策として `*.ps1` スクリプトを介した起動パターンが有効。
- **Linux (WSL2/Ubuntu):** `tmux new-session -d -s <name> <command>` または `nohup <command> &` で直接バックグラウンドプロセスを起動可能。
- **本番サービス:** systemd サービスとして管理されている場合は `sudo systemctl start/stop/restart <service>` を使用する。
- 検証コマンドでプロセス起動を伴うタスクをJulesに依頼する場合は、環境制約を明記すること。

## ツール使用の基本ルール

- ファイル編集は原則として Edit/Write ツールを使用する。`sed`, `awk`, `echo >` 等の bash でのファイル操作は避ける。
- 検索は Glob（ファイル名検索）、Grep（内容検索）を優先し、`find`, `grep`, `cat` の生コマンドは使わない。
- bash 実行時は `description`（5-10語の説明）と `hasSideEffect`（書き込み/状態変更の有無）を必ず設定する。
- タスク完了時は必ずコミット＆プッシュをセットで行う。

## x_search MCP サーバーの使用ルール

このプロジェクトは xAI の Responses API（x_search ツール）を MCP サーバーとしてラップしている。
bash から直接 MCP サーバーを起動して検索する方法：

```
printf '{"jsonrpc":"2.0","id":1,"method":"initialize"}\n{"jsonrpc":"2.0","id":2,"method":"tools/call","params":{"name":"x_search","arguments":{"query":"検索クエリ","max_results":5}}}\n' | timeout 30 node dist/index.js 2>/dev/null | python3 -c "import json,sys
for line in sys.stdin:
    try:
        d=json.loads(line)
        if 'result' in d and 'content' in d['result']:
            print(d['result']['content'][0]['text'])
    except: pass"
```

### 絶対ルール
- **x_search を実行したら、結果の全文を必ずユーザーにメッセージとして貼ること。** 「要約して伝える」「結果をファイルに書く」だけではダメ。ユーザーは検索結果を直接見たい。
- **ユーザーに「結果を貼って」と言われたら、再実行せずに直近の実行結果を使うこと。** 無駄なAPI呼び出し（＝コスト）を発生させてはいけない。
- 認証方式は `XAI_API_KEY`（従量課金）と `XAI_OAUTH_TOKEN`（X Premium）の2種類をサポート。優先順位は OAuth → APIキー。

## コーディングガイドライン

### 0. 実装前に考える（Think Before Coding）

**勝手に前提を置かない。疑問を隠さない。トレードオフを明示する。**
実装を始める前に：
- 自分の仮定を明確に表明する。確証がない場合は必ずユーザーに質問すること。
- 複数の解釈がある場合は勝手に選ばず、すべて選択肢として提示する。
- よりシンプルなアプローチがある場合はその旨を伝える。
- 不明瞭な点がある場合は手を止め、何が混乱しているかを特定して質問する。

### 1. 外科手術のような最小限の変更（Surgical Changes）

**必要な箇所のみを触る。自分で散らかした箇所のみを片付ける。**
- 編集箇所の隣にある無関係なコード、コメント、フォーマットを「改善」しようとしない。
- 壊れていないものをリファクタリングしない。
- 既存のスタイルを忠実に模倣する（自分のやり方と違っていても合わせること）。
- 無関係なデッドコードに気づいたらユーザーに伝えるだけで、勝手に削除しない。
- 自分の変更で使われなくなったインポート、変数、関数は必ず削除する。

### 2. ゴール駆動型の実行（Goal-Driven Execution）

**成功基準を定義する。検証されるまでループする。**
タスクを検証可能なゴールに変換する：
- 「バリデーションを追加する」 ➡ 「無効な入力に対するテストを書き、それをパスさせる」
- 「バグを修正する」 ➡ 「バグを再現するテストを書き、それをパスさせる」
- 「Xをリファクタリングする」 ➡ 「リファクタリングの前後でテストがパスすることを確認する」

複数ステップのタスクではTODOチェックリストを作成する：
```
- [ ] ステップ1: ... ➡ 検証: [チェック内容]
- [~] ステップ2: ... ➡ 検証: [チェック内容]  ← 進行中
- [x] ステップ3: ... ➡ 検証: [チェック内容]  ← 完了
- [-] ステップ4: ... ➡ 検証: [チェック内容]  ← 保留
```
- キリの良い単位でTODOを更新し、完了済み項目と残作業を明確にする
- 設計判断が必要になった場合は、判断理由と採用案を明記してから続行する

### 3. テストファーストの検証

- コード変更後は必ず既存テストが通ることを確認する。
- バグ修正時は再現テストを先に書き、それをパスさせることで修正を確認する。
- 新しい機能には対応するテストを追加する。
