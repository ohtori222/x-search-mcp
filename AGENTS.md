人格定義・応答スタイルの詳細は `PERSONA.md` を参照すること。このファイルが唯一の正規の人格定義であり、以下のルールは補足的な行動制約のみを記述する。

行動の絶対ルール（自爆防止）:
- 【重要】OpenCode（あなた自身）は Node.js 上で動作しています。そのため、プロセスを停止するコマンド（`killall node` など）を実行すると、自分自身も強制終了してしまいます。
- Node.js のプロセスを停止・再起動する必要がある場合は、決して一括キルコマンドを使わず、ポート番号を指定して対象のプロセスID（PID）のみを特定し、個別に停止してください。
- 【重要】**破壊的な削除コマンド（`rm -rf` など）の実行は厳禁です。**
  - プロジェクトルート全体や、絶対パスのルート（`/`）を指定した削除は絶対に実行してはいけません。
  - ファイルやフォルダを削除する必要がある場合は、自分がそのセッションで作成した一時ファイルや、限定された特定の安全なファイルのみを、1つずつ個別に削除してください。大規模な「整理」や「一括クリーンアップ」は絶対に行わず、ユーザーに確認してください。

## 秘密情報の保護（Zero-Trust Secret Protection）
**`.env` ファイルを直接編集・新規作成してはいけません。**
- シークレット（APIキー、トークンなど）をコードや `.env` に直接書き込んでハードコードすることは絶対に禁止です。
- もし環境変数の検証や設定変更が必要な場合は、環境変数の取得ロジック（`process.env` や `os.getenv` など）のみを記述し、実際の値の設定はユーザーに「環境変数に値を設定してください」と手動でお願いしてください。

## コーディングガイドライン

### 1. 実装前に考える（Think Before Coding）
**勝手に前提を置かない。疑問を隠さない。トレードオフを明示する。**
実装を始める前に：
- 自分の仮定を明確に表明する。確証がない場合は、必ずユーザーに質問すること。
- 複数の解釈がある場合は、勝手に選ばず、すべて選択肢として提示する。
- よりシンプルなアプローチがある場合は、その旨を伝える。必要に応じて別の提案をすることも厭わない。
- 不明瞭な点がある場合は手を止め、何が混乱しているかを特定してユーザーに質問する。

### 2. 外科手術のような最小限の変更（Surgical Changes）
**必要な箇所のみを触る。自分で散らかした箇所のみを片付ける。**
既存のコードを編集するとき：
- 編集箇所の隣にある無関係なコード、コメント、フォーマットを「改善」しようとしない。
- 壊れていないものをリファクタリングしない。
- 既存のスタイルを忠実に模倣する（自分のやり方と違っていても合わせること）。
- 無関係なデッドコードに気づいた場合は、ユーザーに伝えるだけで、勝手に削除しない。
自分の変更によって未使用になったもの：
- 自分の変更によって使われなくなったインポート、変数、関数は必ず削除する。
- 自分が触る前から存在していたデッドコードは、指示がない限り削除しない。

### 3. ゴール駆動型の実行（Goal-Driven Execution）
**成功基準を定義する。検証されるまでループする。**
タスクを検証可能なゴールに変換する：
- 「バリデーションを追加する」 ➡ 「無効な入力に対するテストを書き、それをパスさせる」
- 「バグを修正する」 ➡ 「バグを再現するテストを書き、それをパスさせる」
- 「Xをリファクタリングする」 ➡ 「リファクタリングの前後でテストがパスすることを確認する」

複数ステップのタスクでは、TODOチェックリストを作成する：
```
- [ ] ステップ1: ... ➡ 検証: [チェック内容]
- [~] ステップ2: ... ➡ 検証: [チェック内容]  ← 進行中
- [x] ステップ3: ... ➡ 検証: [チェック内容]  ← 完了
- [-] ステップ4: ... ➡ 検証: [チェック内容]  ← 保留
```
- キリの良い単位でTODOを更新し、完了済み項目と残作業を明確にする
- 設計判断が必要になった場合は、勝手に曖昧なまま進めず、判断理由と採用案を明記してから続行する
- 各作業セクションの終わりには「今回の進捗」「確認できたこと」「次の予定」を報告する

明確な成功基準を持つことで、自律的にループして検証する。曖昧な基準（「動くようにする」など）は、常に明確化を求めること。
